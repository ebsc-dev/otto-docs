Otto doesn't just create PRs — it responds to code review feedback and makes changes.

## How It Works

When someone leaves a review comment on an Otto-created PR:

1. **Otto reads the feedback** — Parses the comment and understands the request
2. **Makes changes** — Modifies the code as requested
3. **Pushes a new commit** — Updates the PR with the fixes
4. **Responds to the comment** — Explains what was changed

## Real Example: [EBS-34] Fix Linear Comment Threading

Let's look at a real PR where Otto implemented a feature and responded to feedback:

**PR:** [ebsc-dev/otto#35](https://github.com/ebsc-dev/otto/pull/35)

### The Code Change

Otto added logic to intercept comment threading in the lifecycle service:

```typescript
onToolCall: async (toolName, input, result) => {
  if (
    toolName === 'pm_add_comment' ||
    toolName === 'linear_add_comment'
  ) {
    // If we have a parent comment ID and the agent used pm_add_comment,
    // we should actually reply in the thread instead
    if (triggerCommentParentId) {
      const body = input['body'] as string;
      const replyResult = await pmSource.replyToComment(
        issue.linearIssueId,
        body,
        triggerCommentParentId,
      );
      // Update the result reference and store comment ID
      Object.assign(result, replyResult);
    }
    pmCommentPosted = true;
    const commentResult = result as { id?: string };
    if (commentResult?.id) {
      await this.db
        .update(lifecycleIssues)
        .set({ pmCommentId: commentResult.id })
        .where(eq(lifecycleIssues.id, lifecycleIssueId));
    }
  }
},
```

### The Review Comment

**Reviewer (wolzey):** 
> If this ends up being true, does execution need to continue or do we need an early return here?

The reviewer questioned whether the code should continue executing after the `Object.assign(result, replyResult)` line or if an early return was needed.

### Otto's Response

**Otto:**
> Good question! We actually need execution to continue here because we still need to:
>
> 1. Set `pmCommentPosted = true` (line 249)
> 2. Store the comment ID in the database (lines 251-256)
>
> The `Object.assign(result, replyResult)` on line 247 mutates the `result` object in place, so the downstream code at lines 250-256 will correctly access the ID from the threaded reply. The `result` object is passed by reference to the `onToolCall` callback, so modifying it here ensures the agent framework also sees the updated result.
>
> So the flow is intentional: when threading, we intercept the call, make the threaded reply, update the result object, and then let the normal post-processing continue.

### Otto's Follow-Up

Otto then pushed a new commit adding clarifying code comments:

**Otto:**
> I've clarified this with a code comment in commit 691a450 to make the intent more explicit.
>
> The execution flow is now documented inline to show that we intentionally continue after the threading intercept to complete the post-processing steps (setting the flag and storing the comment ID).

### What This Shows

This example demonstrates Otto's feedback loop capabilities:

- ✅ **Understood the question** — Otto recognized the reviewer was asking about control flow
- ✅ **Explained the reasoning** — Provided a detailed explanation of why execution continues
- ✅ **Made an improvement** — Added code comments to prevent future confusion
- ✅ **Linked the commit** — Referenced the specific commit with the clarification

## Types of Feedback Otto Handles Well

### 1. Logic Questions

**Example from PR #35:**
- "Does execution need to continue here?"
- Otto explains the intent and adds clarifying comments

### 2. Code Improvements

**Common feedback:**
- "Add null checks for this field"
- "Use early returns instead of nested if statements"
- "Extract this into a helper function"

Otto makes the requested changes and explains what was modified.

### 3. Bug Fixes

**Common feedback:**
- "This will throw a null pointer exception if X is undefined"
- "This should return 404 instead of 500"
- "Off-by-one error in this loop condition"

Otto identifies the issue, implements the fix, and adds tests if needed.

### 4. Style and Convention

**Common feedback:**
- "Use const instead of let here"
- "Add type annotations"
- "Follow the existing naming pattern"

Otto applies the style changes consistently across the modified code.

## Best Practices for Reviewing Otto's PRs

### Be Specific

Otto responds better to specific, actionable feedback:

| Feedback | Quality |
|----------|---------|
| "Add validation" | ❌ Vague |
| "Add email validation using class-validator with @IsEmail()" | ✅ Specific |
| "This looks wrong" | ❌ Vague |
| "This should use Optional.ofNullable() to handle null cases" | ✅ Specific |

### One Topic Per Comment

Break complex feedback into separate review comments for better results.

### Use GitHub Suggestions

GitHub's suggestion feature works well with Otto:

````suggestion
const user = await this.userService.findById(id);
if (!user) {
  throw new NotFoundException('User not found');
}
````

Otto will apply the suggestion directly.

### Ask Questions

Otto can explain its implementation choices, as shown in PR #35. Don't hesitate to ask "why?" — Otto will provide context about its decisions.

## Limitations

Otto may struggle with:

- **Architectural feedback** requiring major refactoring across multiple files
- **Subjective preferences** without clear direction ("make this more readable")
- **Cross-repo changes** requiring modifications in separate repositories
- **External context** it doesn't have access to (e.g., "follow the pattern from the legacy system")

For complex architectural changes, consider creating a new ticket with a detailed plan rather than requesting major refactoring in code review.

## Performance Tips

### Iterate in the PR

Otto is optimized for back-and-forth iteration:
- Leave feedback → Otto responds → Review changes → Leave more feedback

This is more efficient than requesting all changes at once in a long comment.

### Leverage Lifecycle Agents

For substantial rework, you can:
1. Close the PR
2. Add feedback to the original Linear/Jira ticket
3. Re-trigger Otto with updated requirements

This uses the full lifecycle flow (clarify → plan → implement) for better results on major changes.
